High‑priority findings (immediate fixes)
Weak refresh token handling (plaintext storage, non-cryptographic generation, no rotation)

Risk: If DB is leaked or attacker obtains tokens, tokens can be used to impersonate users. No rotation/rehash/reuse detection increases blast radius.
Fixes:
Generate refresh tokens using a secure RNG (e.g., 32+ bytes from RandomNumberGenerator) and treat them as opaque.
Store only a hashed version (SHA-256) of the refresh token server-side; compare using hash.
Implement refresh token rotation: when a refresh request is used, issue a new refresh token and mark the old one revoked; support token family and detect reuse to revoke family.
Add fields to RefreshToken entity: TokenHash, TokenFamilyId, ReplacedByTokenId, RevokedAtUtc, IpAddress, UserAgent. Stop storing Token plaintext.
Update AuthRepository API to lookup by token hash, not plaintext.
Why: Prevents disclosure of usable tokens from DB and supports robust session management.
Refresh endpoint does not rotate or revoke refresh tokens

Risk: Long-lived replayable refresh tokens.
Fixes:
In AuthController.Refresh and the refresh handler: validate token, create a new refresh token, store its hashed value, set ReplacedByTokenId and revoke or mark the used token, return the new refresh token to client.
If reuse of revoked token is detected, revoke entire token family and force re-login.
JWT secret fallback and key management

Risk: Hard-coded fallback "dev-secret-key-please-change" is insecure; weak keys allow token forgery.
Fixes:
Require secret in configuration; fail startup if missing.
Use strong secret (minimum 32+ chars), rotate regularly.
Use secure secret storage (Azure Key Vault, AWS Secrets Manager, environment variables injected by CI/CD).
Set ValidateIssuerSigningKey = true explicitly and set small ClockSkew (e.g., 30s).
Refresh tokens returned to client in response body (no HttpOnly cookie)

Risk: If frontend stores refresh token in localStorage it is exposed to XSS; better to keep refresh tokens in HttpOnly, Secure, SameSite cookies or use rotating refresh-token cookie + anti-CSRF.
Fix:
Return access token in response body, but set refresh token in an HttpOnly, Secure, SameSite cookie. Use anti-CSRF protections for cookie-based flows (double submit cookie or same-site + CSRF tokens).
If you must use persisted JS storage, educate about XSS risk and implement additional mitigations.
